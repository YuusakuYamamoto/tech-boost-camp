# ADR-0004: 単一 VCN によるネットワーク基盤

- **Status**: Proposed
- **Date**: 2026-04-25
- **Deciders**: Yusaku

## Context

本アプリのインフラリソース（Container Instance、PostgreSQL VM、LB）を配置するためのネットワーク基盤を設計する必要がある。本アプリは個人プロジェクトで、他のアプリと同居する予定はないが、

- パブリック / プライベートのサブネット分離
- セキュリティグループ・経路制御
- 外部送信用ゲートウェイ（Service Gateway / NAT Gateway）

の構成を決める必要がある。

## Decision

**単一 VCN** を立て、その中に必要なサブネット・ゲートウェイを配置する。

### 構成

| リソース             | 内容                                                       |
| -------------------- | ---------------------------------------------------------- |
| VCN                  | 1 つ（CIDR 例: 10.0.0.0/16）                               |
| パブリックサブネット | LB を配置（CIDR 例: 10.0.1.0/24）                          |
| プライベートサブネット | Container Instance / PostgreSQL VM を配置（10.0.2.0/24） |
| Internet Gateway     | LB がパブリック IP で受信するため必要                      |
| Service Gateway      | OCIR / Vault / Object Storage / Oracle Linux ミラーへの経路 |
| NAT Gateway          | **不要**（外部 API 連携なし、OS パッチは Service Gateway 経由） |

NAT Gateway を立てない設計にすることで、月 $32 の固定費を削減する。これは Oracle Linux + Service Gateway の組み合わせ（ADR-0018）で実現する。

## Alternatives Considered

### NAT Gateway を立てて、外部送信を可能にする

- **不採用理由**:
  - 月 $32 の固定費がコスト目標（数ドル/月）に対して大きすぎる
  - そもそも外部 API 連携を想定していないため、用途がない
  - OS パッチ適用は Service Gateway 経由で OCI 内のミラーから取得可能（Oracle Linux 採用前提）

### Container Instance / VM をパブリックサブネットに直接配置

- **不採用理由**:
  - インスタンスに直接インターネットからアクセス可能になり、セキュリティリスクが増加
  - LB を前段に置く意義が薄れる

## Consequences

### Positive

- NAT Gateway 不要のためコストを大幅削減
- 一般的な「LB はパブリック / アプリ・DB はプライベート」のパターンを踏襲しており、セキュリティ的にも素直
- Service Gateway は無料、OCI 内サービス（OCIR、Vault、Object Storage、Oracle Linux ミラー）への通信が完結

### Negative / Trade-off

- 外部 API（Google API、SaaS API など）を呼び出したくなった場合は NAT Gateway を後付けで立てる必要がある
- OS は Oracle Linux に固定される（Ubuntu 等を使うなら別途 NAT Gateway が必要）

### Neutral

- VCN の CIDR は将来的にサブネットを追加できるよう /16 で確保しておく
