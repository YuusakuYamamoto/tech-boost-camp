# ADR-0017: 監視・アラートの初期方針

- **Status**: Proposed
- **Date**: 2026-04-25
- **Deciders**: Yusaku

## Context

運用開始時から以下を把握できる状態にしておく必要がある：

- アプリケーションが正常に動作しているか
- インフラリソースに異常がないか
- 想定を超える費用が発生していないか（個人アプリのコスト目標は月数ドル）

最初から精緻な監視・アラート設計を行うコストは規模に見合わない。本格的な監視・ログ分析は運用開始後の状況を見ながら拡充していく。

## Decision

最初の運用フェーズでは、**OCI 標準の Logging / Monitoring サービスを使い、最低限の運用アラートとコストアラートのみ設定する**。

### 設定する最低限のアラート

| アラート                          | 閾値の目安                              | 通知先         |
| --------------------------------- | --------------------------------------- | -------------- |
| LB ヘルスチェック失敗（全滅）     | OK 状態の Backend = 0 が 3 分継続        | 即時通知       |
| Container Instance が停止状態     | desired と異なる状態を検知              | 即時通知       |
| PostgreSQL VM のディスク使用率    | 80% 超過                                | 即時通知       |
| PostgreSQL コンテナの異常停止     | コンテナの health 失敗                  | 即時通知       |
| 月間予算アラート                  | 予算（$10）の 50% / 80% / 100%          | メール         |
| OCI API の異常な使用量            | 通常の数倍以上の呼び出し                | メール         |

### ログ取得

OCI Logging の標準設定で以下を有効化する：

- Container Instances の標準出力（stdout/stderr）
- LB アクセスログ
- PostgreSQL VM の主要ログ（OS / コンテナ）

### コスト保護

OCI Budget 機能で本アプリの Compartment に予算（$10/月）を設定する。Container Instance や VM の誤った構成、想定外のコスト発生を早期検知する。

## Alternatives Considered

### Datadog / New Relic 等の商用 SaaS

- **不採用理由**:
  - 個人アプリの規模・コスト目標に対して過剰
  - OCI 標準のサービスで初期要件は満たせる

### 監視は完全に後回し

- **不採用理由**:
  - 異常検知の仕組みなしで運用するのはリスク
  - コストの暴走を防ぐ Budget アラートは特に重要（個人課金のため）

## Consequences

### Positive

- 初期コストを最小化しつつ、最低限の異常検知ができる
- OCI 標準サービスのため、追加の運用負担なし
- Budget アラートで誤設定によるコスト事故を早期検知できる（個人課金保護）

### Negative / Trade-off

- アプリケーション層の詳細な性能監視（APM）は含まれない
- ログの相関分析や可視化は限定的
- 通知先がメールのみ

### Neutral

- 運用開始後、必要性に応じて以下を追加検討する：
  - APM ツール導入
  - 構造化ログと Logging Analytics の活用
  - SLO / SLI の定義
