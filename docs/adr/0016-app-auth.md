# ADR-0016: アプリケーション認証は Google アカウント + allowlist

- **Status**: Proposed
- **Date**: 2026-04-25
- **Deciders**: Yusaku

## Context

本アプリは認証付きパブリック公開（仲間内利用）。利用ユーザーは事前にアドレスを聞いて allowlist に登録した人のみログイン可能とする。

要件：

- 自分（管理者）と一般ユーザー（仲間）の 2 つのロールを区別する
- ユーザー識別子は変更耐性のあるものを使う
- 認証情報（パスワード等）の自前管理は避けたい

## Decision

PoC アプリと同じく以下の方針で実装する：

| 項目                  | 方針                                                                |
| --------------------- | ------------------------------------------------------------------- |
| プロトコル            | OIDC（Authorization Code Flow + PKCE）                              |
| OAuth クライアント    | 個人の GCP プロジェクトで作成                                       |
| ユーザー識別子        | Google `sub`（subject）。メールアドレスは表示用                     |
| ログイン制御          | メールアドレス allowlist 方式。DB のユーザーテーブルに事前登録      |
| ロール管理            | DB で管理。`users.role` カラムで `admin` / `member` を表現          |
| 管理者の決定          | 自分のメールアドレスを `admin` で初期投入                           |
| セッションタイムアウト | 短めに設定（例: 8 時間）                                            |

## Alternatives Considered

### メールアドレスをユーザー識別子にする

- **不採用理由**:
  - メールアドレス変更時に履歴が壊れる
  - `sub` を主キーにし、メールアドレスを表示用カラムにする方が変更耐性がある

### 認証なしで公開（自分しか URL を知らない前提）

- **不採用理由**:
  - URL 漏洩時に第三者がアクセス可能
  - DB に保存する情報の性質によっては、認証なし公開は不適切

### 自前のメール + パスワード認証

- **不採用理由**:
  - パスワードハッシュ管理・パスワードリセット機能・MFA 実装などの複雑性
  - Google アカウントログインの方が実装が容易で、ユーザー側の負担も少ない

## Consequences

### Positive

- 認証情報の管理を Google に委譲できる
- 仲間のアドレスを事前登録するだけで利用可能
- ロール管理が DB で完結、追加要件にも柔軟に対応できる
- ユーザー識別子・ロール・セッションの設計が明確

### Negative / Trade-off

- allowlist のメンテナンス（新規ユーザー追加時）が必要
- セッションタイムアウトを短めに設定するため、ユーザーの再ログイン頻度が上がる
- Google アカウントを持たない人は利用できない（仲間内利用なら問題にならない）

### Neutral

- GCP プロジェクトは個人で作成・管理する
- 認証ライブラリは `passport-google-oauth20` 等の標準的なものを使う
- OAuth クライアントシークレットは Vault に保管（ADR-0014 / ADR-0015）
